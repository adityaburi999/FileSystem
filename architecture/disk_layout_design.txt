DISK LAYOUT DESIGN (CURRENT DEMO)
========================================================
/storage_root
========================================================

/wal/
    wal.log
- Append-only WAL file
- Source of truth for transaction replay on restart

========================================================
/metadata/
========================================================

Option A (file-backed metadata backend):
/metadata/
    metadata.json

Option B (sqlite metadata backend):
/metadata/
    metadata.db
    metadata.db-wal
    metadata.db-shm

- Stores file state (version, chunk refs, tombstone)
- Stores namespace directories for mkdir/readdir/rmdir
- Stores applied transaction IDs for idempotent replay

========================================================
/objects/
========================================================

/objects/
    <chunk-id>.chunk (or sharded form by backend)

- Immutable content-addressed chunks
- Chunk ID is BLAKE3 hash of content
- Never overwritten in place

========================================================
/cache/
========================================================

In-memory mode:
- L1 + L2 both in memory

Persistent warm-tier mode:
/cache/
    <fingerprint>.json

- Warm tier entries persisted on disk
- Reloaded on process restart
- Path invalidation and full invalidation supported

========================================================
/staging/
========================================================

/staging/
    <tx-id>/
        slot.meta

- Per-transaction temporary write slot
- Marks committed vs uncommitted
- Startup purge removes stale uncommitted slots

========================================================
/system/
========================================================

/system/gc/
- Scheduler state and worker lifecycle owned in process memory
- Optional persisted files can be added later

========================================================
KEY RULES (CURRENT)
========================================================

1. WAL remains append-only and replayable.
2. Metadata is authority for visible namespace + file versions.
3. Objects are immutable by chunk ID.
4. Cache is performance layer; warm-tier persistence is optional.
5. Staging is never visible to namespace reads.
6. Mount opens only after successful recovery.
