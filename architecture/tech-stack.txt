## MetaData Engine
    Optimizations Needed-
        - Sub MilliSecond Lookups
        - Strict write ahead safety(Acid Compliance)
        - Low memory overhead
    Tech Stack- 
        - Rust : core metadata logic
        - SQLite: Primary MetaData Database, helps achive this goal perfectly.
        - Serde: Serialization framework for Rust
        - Bincode: Compact binary Serialization for fast metadata storage.
        - BLAKE3: 
        - WAL: Used to prevent metadata corruption during crashes.
        - Copy-on-write MetaData: Metadata updates creates new versions instead of modifying existing structures directly. Improves crash safety and rollback cabability  

    Future Migration to RocksDB/custom metadata Engine
## Storage Backend
    - Content-addressable object storage: Files are stored based on content hashes instead of physical paths.
    - Chunk-based storage architecture: Large files split into chunks.
    - Immutable blob objects: Stored chunks never modified directly.
    - BLAKE3 content hashing: 
        Used for:
            - chunk identity
            - corruption detection
            - deduplication
            - integrity verification
    - Copy-on-write storage model: Modified data creates new storage objects, it created partial corruption during crashes
    - Inline small-file optimization: Tiny files stored directly inside metadata objects instead of separate backend blobs.
    - Compressed object containers
    - Background garbage collection: Unused chunks cleaned asynchronously.
