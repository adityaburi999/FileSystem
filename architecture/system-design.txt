## Read File (Current)
1. Request arrives through FuseApi or FuseDaemon.
2. Path is canonicalized.
3. Core checks mount gate (must be open after startup recovery).
4. Metadata read returns committed chunk list/version.
5. Cache lookup: L1 then warm tier.
6. On miss, chunks are read from object store.
7. Chunk hash is validated before cache fill.
8. Bytes are assembled and range-sliced for response.

## Write/Create File (Current)
1. Request arrives with expected metadata version.
2. Core opens staging slot (optional staging manager).
3. Pipeline streams bytes into fixed-size chunks.
4. Each chunk ID is BLAKE3(content); chunk persisted immutable.
5. WAL records pending transaction and chunk refs.
6. Metadata commit uses CAS expected_version check.
7. WAL commit marker is appended.
8. Core post-commit: staging cleanup + cache invalidate(path).

## Delete File (Current)
1. Request arrives with expected metadata version.
2. Core verifies file exists and is not a directory target for unlink_checked.
3. WAL begin delete transaction.
4. Metadata tombstone commit with CAS.
5. WAL commit marker.
6. Cache invalidate(path) and GC enqueue (best effort).

## Namespace Rules (Current)
- mkdir(path): parent directory must exist.
- readdir(path): directory must exist, else not found.
- rmdir(path): only empty directory can be removed.
- rmdir/rmtree on file path: rejected as conflict (path-type mismatch).
- write(path): parent directory must exist and path must not be a directory.
- mkdir(path): path must not be an existing live file.
- unlink(path): directory path is rejected as conflict.

## Startup Recovery (Current)
1. WAL scanned in append order by latest tx state.
2. Committed writes are validated and replayed into metadata.
3. Pending transactions are marked aborted.
4. Cache invalidate_all.
5. Staging purges stale slots (uncommitted first, then any leftover committed slots).
6. Mount gate opens only after successful recovery.

## GC Model (Current)
- Trigger enqueue is non-blocking.
- Trigger enqueue backlog is bounded (default max pending scans = 64) to avoid unbounded queue growth under churn.
- Demo/runtime knobs can tune that queue bound where GC builder wiring is exposed.
- Worker scans all chunks and removes orphan candidates.
- Inflight WAL refs are protected from deletion.
- Retention window delays deletion of newly orphaned chunks.
- Optional per-sweep delete budget limits deletion burst size.
- Periodic scheduler can run trigger + worker loop.
- Demo runtime can switch basic flow from one-shot GC drain to periodic scheduler mode via env knob.

## Failure Mapping (Current)
- CAS mismatch -> conflict/busy.
- Missing parent directory -> not found.
- Directory not empty on rmdir -> conflict/not empty.
- Recovery error -> mount remains closed.
- Integrity mismatch -> IO/integrity error.

## Concurrency Model (Current)
- Metadata writes are optimistic CAS.
- Delete has bounded retry with backoff on CAS conflicts.
- Read path is lock-light except shared cache metadata/state.
- GC is background and best-effort relative to foreground correctness.

## Current Limitations (Intentional for demo)
- No kernel FUSE mount/syscall integration yet; current fuse crate is API + daemon loop.
- No distributed storage/sharding.
- No production observability stack.
